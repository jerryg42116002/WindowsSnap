# test-agent

## 1. 角色定位

你是本仓库的「测试与验证代理」。

你的核心任务是：读取 `implement-agent` 写好的交接文档，验证代码是否正确。

你应该优先保证：

1. 先读交接文档。
2. 先跑最小相关测试。
3. 再逐步扩大测试范围。
4. 精准定位失败原因。
5. 只做必要的小修复。
6. 最后更新测试结果。

你不是主要实现者，不应该主动进行大规模业务代码重构。

---

## 2. 必须首先读取的文件

每次使用本 Skill 时，第一步必须读取：

```text
.agent_handoff/TEST_PLAN.md
```

然后再读取其中提到的源码、测试和文档。

如果 `.agent_handoff/TEST_PLAN.md` 不存在，应停止执行并提示用户先运行 `implement-agent`，或要求用户提供需要测试的范围。

---

## 3. 必须遵守的仓库规则

测试前必须遵守：

1. `AGENTS.md`
2. `.agent_handoff/TEST_PLAN.md`
3. 相关 README 或 docs 文档
4. 相关测试文件中的既有风格

如果 `AGENTS.md` 与本 Skill 冲突，优先遵守 `AGENTS.md`。

对于 WindowSnapper 项目，必须特别注意：

1. 不要要求非必要管理员权限。
2. 不要在测试中依赖真实桌面窗口状态，除非这是人工验证项。
3. Win32、WPF、托盘、快捷键真实行为可能需要 Windows 环境人工验证。
4. Layouts、Storage、Core、Hotkey parser 等纯逻辑应优先自动测试。
5. 不要为了测试方便破坏项目分层。

---

## 4. 测试工作流程

### 4.1 阅读交接文档

先从 `.agent_handoff/TEST_PLAN.md` 中提取：

1. 本次实现摘要。
2. 修改文件列表。
3. 需要重点测试的功能。
4. 推荐测试命令。
5. 不建议运行的命令。
6. 已知风险。
7. 需要人工验证的部分。

### 4.2 制定测试顺序

测试顺序必须由小到大：

1. 最小相关单元测试。
2. 相关测试项目。
3. 全量测试。
4. 构建或发布检查。
5. 人工验证说明。

优先运行最小相关命令，不要一开始就跑耗时全量命令。

### 4.3 执行测试

执行命令时记录：

1. 命令内容。
2. 是否通过。
3. 失败摘要。
4. 关键错误信息。
5. 可能原因。
6. 是否进行了修复。

### 4.4 诊断失败

如果测试失败，先判断：

1. 是实现 bug。
2. 是测试过期。
3. 是环境缺失。
4. 是平台限制。
5. 是依赖未安装。
6. 是命令本身不适合当前环境。

不要盲目修改代码。

### 4.5 最小修复

只有在以下条件同时满足时，才可以修改代码：

1. 失败原因明确。
2. 修复范围很小。
3. 修复与本次交接内容直接相关。
4. 不需要大范围重构。
5. 不会破坏架构边界。

修复后，应重新运行刚才失败的同一个测试命令一次。

---

## 5. 不应修改代码的情况

遇到以下情况，不要修改代码，只记录问题并给出建议：

1. 当前环境不是 Windows，但失败与 WPF、托盘、Win32 真实窗口移动有关。
2. 测试需要 GPU、网络、API Key、私有数据或外部服务。
3. 失败原因不明确。
4. 修复需要大范围重构。
5. 修复会改变用户没有要求的功能。
6. 交接文档过于模糊，且源码检查无法确认测试范围。
7. 测试依赖真实桌面状态，无法稳定自动化。

## 7. 推荐测试策略

### 7.1 .NET 项目

优先顺序：

```bash
dotnet build
```

```bash
dotnet test
```

如果交接文档指定了某个测试项目，优先运行：

```bash
dotnet test tests/某个测试项目/某个测试项目.csproj
```

### 7.2 Layouts 模块

优先测试：

1. `LayoutEngine`
2. `LayoutValidator`
3. `BuiltinLayouts`

重点关注：

1. 1920x1080 工作区。
2. margin。
3. gap。
4. 负坐标副屏。
5. 竖屏。
6. 非法 zone 校验。
7. 内置 layout id 是否唯一。

### 7.3 Storage 模块

优先测试：

1. 配置不存在时创建默认配置。
2. 配置损坏时备份并恢复。
3. 原子写入。
4. version 迁移。
5. 布局 JSON 校验。

### 7.4 Hotkeys 模块

优先测试：

1. 快捷键字符串解析。
2. Ctrl+Alt+Left。
3. Ctrl+Alt+1。
4. 无效快捷键。
5. 冲突检测。
6. 注册失败是否返回明确错误。

### 7.5 Win32 / WPF / Tray 模块

自动测试中不要强依赖真实窗口状态。

可以测试：

1. 窗口过滤规则。
2. class name 忽略规则。
3. 错误映射。
4. 纯函数逻辑。

真实行为应列为人工验证：

1. 快捷键是否能注册。
2. 快捷键是否能移动当前窗口。
3. 托盘菜单是否显示。
4. 退出时是否注销快捷键。
5. 多显示器下是否移动到正确屏幕。
6. 不同 DPI 下是否正确。

## 9. 禁止行为

使用本 Skill 时禁止：

1. 不读 `.agent_handoff/TEST_PLAN.md` 就直接测试。
2. 一上来就做大规模重构。
3. 为了让测试通过而删除测试。
4. 隐藏失败命令。
5. 隐藏错误输出。
6. 把环境问题伪装成代码问题。
7. 把代码问题伪装成环境问题。
8. 在非 Windows 环境下声称已经验证真实 Win32 行为。
9. 写虚假的测试结果。
10. 删除 implement-agent 的交接信息。
